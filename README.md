# Velora

Velora — лёгкая серверная основа на Go для мультиплеерных игр с авторитарной серверной моделью.

На текущем этапе проект реализует:

- HTTP-сервер с WebSocket endpoint;
- регистрацию клиентов внутри центрального хаба;
- автоматическую выдачу уникальных идентификаторов клиентам;
- обмен пакетами в бинарном формате через Protocol Buffers;
- широковещательную доставку сообщений между подключёнными клиентами.

Репозиторий пока небольшой и ближе к прототипу транспортного слоя, чем к полноценному игровому фреймворку, но базовый сетевой контракт уже определён.

## Возможности

- WebSocket endpoint at `/velora`
- Настраиваемый порт запуска через CLI-флаг
- Protobuf-схема пакетов для обмена между клиентом и сервером
- Управление подключениями через хаб
- Рассылка пакетов всем клиентам, кроме отправителя

## Стек

- Go `1.26`
- `gorilla/websocket`
- `google.golang.org/protobuf`

## Структура проекта

```text
.
├── main.go                            # точка входа приложения
├── shared/packets.proto               # общая схема пакетов
├── server/pkg/packets                 # сгенерированный protobuf-код и хелперы
└── server/Internal
    ├── idgenerator.go                 # генератор последовательных ID клиентов
    ├── objects/sharedCollection.go    # потокобезопасное хранилище клиентов
    └── server
        ├── hub.go                     # центральный хаб и маршрутизация
        └── clients
            ├── websocket_factory.go   # создание WebSocket-подключения
            ├── websocket_client.go    # инициализация клиента
            ├── websocket_readwriter.go# циклы чтения и записи в сокет
            ├── websocket_route.go     # маршрутизация и broadcast
            └── websocket_close.go     # закрытие соединения
```

## Запуск

Требования:

- Go `1.26+`

Запуск сервера:

```bash
go run . 
```

По умолчанию сервер слушает порт `8080`.

Запуск на произвольном порту:

```bash
go run . -port 9000
```

WebSocket endpoint будет доступен по адресу:

```text
ws://localhost:8080/velora
```

или, если указан другой порт:

```text
ws://localhost:<port>/velora
```

## Формат пакетов

Все данные передаются как бинарные WebSocket-сообщения, сериализованные через Protocol Buffers.

Источник схемы: [`shared/packets.proto`](./shared/packets.proto)

```proto
message Packet {
  uint64 sender_id = 1;
  oneof msg {
    ChatMessage chat = 2;
    IdMessage id = 3;
  }
}
```

Поддерживаемые типы сообщений:

- `ChatMessage` — обычное текстовое сообщение
- `IdMessage` — служебное сообщение с назначенным ID клиента

## Сценарий подключения

1. Клиент подключается к `/velora` по WebSocket.
2. Сервер повышает HTTP-соединение до WebSocket.
3. Хаб регистрирует клиента и назначает ему уникальный идентификатор `uint64`.
4. Клиент получает `IdMessage` со своим ID.
5. Все последующие пакеты клиента рассылаются остальным подключённым клиентам.

Если `sender_id` во входящем пакете равен `0`, сервер заменяет его на ID текущего соединения.

## Разработка

Проверка проекта:

```bash
go test ./...
```

Перегенерация protobuf-кода после изменения схемы:

```bash
protoc --go_out=server shared/packets.proto
```

Перед запуском этой команды убедитесь, что установлены `protoc` и `protoc-gen-go`.

## Текущие ограничения

Текущая реализация намеренно остаётся минимальной. В ней пока нет:

- аутентификации и авторизации;
- комнат, каналов или игровых сессий;
- репликации и синхронизации состояния;
- отдельного слоя надёжности поверх WebSocket-сообщений;
- проверки origin при WebSocket upgrade (`CheckOrigin` сейчас разрешает любые источники);
- отдельного прикладного слоя команд поверх базовой схемы пакетов.
